NON-NEGOTIABLE ARCHITECTURAL CONSTRAINTS

- `ClinicalEvidence` is RCV-centered, not a flat list: submissions
  belong to a specific condition-interpretation (RCV), not to the
  variant as an undifferentiated pool. A variant can be classified
  differently for different conditions — the model must preserve that.
- ClinVar's own aggregate classification and review status (per
  condition) are preserved exactly as the source provides them — never
  recomputed, never voted, never averaged.
- Review status stars are NOT built in this phase (5.5B-2's job) — this
  phase only needs to retrieve and store the raw review status text.
- No `VariantResearchContext` in this phase.
- No PubMed citation mining, no pathway/GO summarization, no population
  frequency, no gnomAD — all remain out of scope per the frozen
  architecture.
- No LLM-generated text anywhere in this phase.
- Never fetch full submission detail for every condition-interpretation
  on initial variant load — lazy, on-demand only (Step 5).
- No new rate limiter, no new generic NCBI client — reuse what already
  exists.
- Never fabricate a condition name, classification, review status, or
  submitter — use `null`/empty array when the source doesn't provide it.

STEP 1 — MANDATORY CLINVAR DEEPER-RETRIEVAL AUDIT

Do this before writing any implementation code. Write findings to
`PHASE-5.5B-1-AUDIT-FINDINGS.md` as you go. This is exactly as
consequential as 5.5A's own Step 1 audit — 5.5A already proved ESummary
alone cannot provide submission-level data, so this phase's entire
retrieval design depends on what you find here, not on assumption.

Determine, with concrete evidence (real API calls, real responses):
1. Does `efetch.fcgi?db=clinvar&rettype=vcv&id={VCV_ACCESSION}` (full
   VCV XML) actually contain RCV-level condition interpretations and
   SCV-level submission data? Test against a real, well-studied variant
   (e.g. a known BRCA1 or TP53 pathogenic variant from 5.5A's own
   smoke-test data).
2. If VCV XML doesn't contain sufficient data, what NCBI surface does?
   Test alternatives systematically — do not stop after checking only
   one format. Explicitly check: XML (`rettype=vcv`), JSON (if a
   `retmode=json` variant exists for this endpoint), a separate
   RCV-level ESummary call, and whether NCBI currently exposes a
   dedicated ClinVar REST API distinct from classic E-utilities (NCBI
   has changed ClinVar's API surface over time — verify what's actually
   live today, don't rely on training-data assumptions about what used
   to exist).
3. **Explicitly rule out ClinVar's bulk FTP flat files** (e.g.
   `variant_summary.txt`, `submission_summary.txt`) as a live,
   per-request retrieval mechanism — these are large periodic bulk
   downloads designed for offline ETL/ingestion pipelines, not
   per-variant on-demand API calls, and using them here would violate
   this phase's on-demand/volume-safe design entirely. Note this
   explicitly in the audit findings so a future session doesn't
   reconsider them as a live-retrieval option. (They remain a
   theoretical future option ONLY for a completely different
   architecture — a scheduled bulk-ingestion job — which is out of
   scope for this phase and not being proposed here.)
4. Can `submissionCount` be obtained cheaply, separate from full
   submission detail? This determines whether the grouped-summary view
   (Section 4/5 of the architecture) can be shown without a full
   submission fetch. **If no native cheap count field exists**, and the
   only way to get a count is to fetch all SCVs anyway, this changes
   the lazy-loading design — document this specifically (see Step 5's
   explicit resolution rule for this exact scenario).
5. Confirm real-world data quirks: multiple RCVs with overlapping/
   near-duplicate condition names, malformed or partial XML sections,
   variants with zero interpretations, variants with a very high
   submission count on a single condition. Document how each will be
   handled. **For overlapping/synonym condition names specifically**
   (e.g. "Breast cancer," "Hereditary breast cancer," "Breast-ovarian
   cancer syndrome" appearing as separate RCVs that may be conceptually
   related): do NOT merge them — preserve ClinVar's raw wording exactly
   as separate `ConditionInterpretation` entries. Capture whatever
   database identifiers ClinVar already provides per condition (MedGen
   CUI, OMIM, MONDO, etc.) in `ClinicalCondition.identifiers` — this
   costs nothing extra (the data is already in the response) and keeps
   the door open for future condition-relationship navigation without
   requiring any merging logic now.
6. Confirm ClinVar's actual review-status text values as they appear in
   real responses (needed later for 5.5B-2's star mapping, but must be
   captured now while you have real data in front of you).

**Decision log requirement:** document every retrieval surface
considered in a table, not just the one chosen:

| Surface | Decision | Reason |
|---|---|---|
| ESummary | ✅/❌ | (e.g. "No SCV data") |
| VCV XML | ✅/❌ | (e.g. "Contains RCV + SCV") |
| RCV-level ESummary | ✅/❌ | ... |
| JSON variant (if any) | ✅/❌ | ... |
| Dedicated ClinVar REST API (if found) | ✅/❌ | ... |
| Bulk FTP files | ❌ | Not suitable for per-request retrieval |

This prevents a future session from re-litigating "why didn't we use
JSON?" with no record of the answer.

**Reference examples requirement:** save a small number of representative
raw API responses (not full dumps — trimmed to the relevant sections) to
`docs/clinvar/example-vcv.xml`, `docs/clinvar/example-rcv-section.xml`,
`docs/clinvar/example-scv-section.xml` (or equivalent, matching whatever
surface Step 1 actually confirms). These become golden reference
material for future debugging and for 5.5B-2's implementation.

Do not proceed to Step 2 until this is complete and documented.

STEP 2 — CLINICAL EVIDENCE DATA CONTRACTS

Based on Step 1's findings, implement (adapt exact shape only if Step 1
reveals a genuine structural need — do not redesign for convenience):

```typescript
interface ClinicalEvidence {
  clinvarVariationId: string;          // links to VariantRecord
  interpretations: readonly ConditionInterpretation[];
}

interface ConditionInterpretation {
  rcvAccession: string;
  conditions: readonly ClinicalCondition[];
  aggregateClassification: string | null;   // ClinVar's own — preserved as-is
  aggregateReviewStatus: string | null;      // ClinVar's own — preserved as-is
  lastEvaluated: string | null;
  submissionCount: number;  // use the native count field if the source
                             // provides one (per Step 1 finding 4); if
                             // no native field exists, compute it from
                             // the actual number of SCVs returned in
                             // the same response — never estimate or
                             // guess this value
  submissions: readonly ClinicalSubmission[]; // may be empty until lazy-loaded, see Step 5
}

interface ClinicalCondition {
  name: string;
  identifiers: readonly { database: string; id: string }[];
}

interface ClinicalSubmission {
  scvAccession: string;
  significance: string;
  reviewStatus: string;
  submitter: string | null;
  lastEvaluated: string | null;
  conditionAsserted: string | null;
}
```

Keep this strictly separate from `VariantRecord` — do not merge fields
into it. Run TypeScript.

STEP 4 — PARSING AND NORMALIZATION

Map the raw ClinVar response into Step 2's contracts. Group submissions

Deterministic failure handling: if one `ConditionInterpretation` fails
to parse, skip it individually while preserving all others successfully
parsed for the same variant — do not abort the whole `ClinicalEvidence`
object.

Run TypeScript.

STEP 5 — LAZY LOADING AND SUBMISSION-LEVEL PAGINATION

- On initial variant selection: fetch `ClinicalEvidence` with
  `aggregateClassification`, `aggregateReviewStatus`, `submissionCount`
  per condition-interpretation — but do NOT eagerly fetch full
  `submissions[]` detail for every condition-interpretation if Step 1
  found these can be split into separate calls. If Step 1 found they
  cannot be split (bundled in one response), document this explicitly
  and proceed with what's actually available — do not force an
  artificial split that doesn't match the API's real shape.

**Explicit rule for the count-vs-fetch-cost tension:** if Step 1 found
that obtaining `submissionCount` requires fetching all SCVs anyway (no
cheap native count field), then the "cost" of showing a count has
already been paid — in that case, do not discard the fetched
submissions just to simulate lazy loading. Store what was already
retrieved and let "expand" simply reveal already-fetched data (a
UI-only expand/collapse, not a new network trigger) rather than
re-fetching. Only treat submission retrieval as a genuinely deferred,
on-demand network call if Step 1 confirmed a cheap, separate count path
actually exists. Document which of these two cases applies.

- Full submission list per condition-interpretation is fetched only on
  explicit user expansion of that specific condition (when a genuinely
  separate on-demand fetch is possible per the rule above), reusing the
  Universal Pagination Framework if `submissionCount` exceeds ~20 (exact
  threshold: adjust based on what real data in Step 1 suggests is
  reasonable, document the chosen value).
- Never fetch submissions for every condition-interpretation on a
  multi-condition variant's initial load, UNLESS Step 1 confirms this is
  unavoidable per the rule above — in which case document it as an
  accepted tradeoff, not an oversight.

Run TypeScript.

FORWARD-LOOKING NOTE (documentation only — do not act on this now)

This phase's retrieval service is ClinVar-specific by design. A future
platform direction may eventually support additional clinical evidence
sources (ClinGen, CIViC, Open Targets, COSMIC, LOVD), at which point an
abstraction like `ClinicalEvidenceProvider` → `ClinVarProvider` might
make sense. Do NOT build this abstraction now — it would be premature
generalization for a single-source phase. Simply avoid naming things in
a way that makes a future provider-interface refactor unnecessarily
painful (e.g. prefer `clinvar-retrieval.ts` over an unqualified
`retrieval.ts` if it reduces ambiguity later). This note is for
awareness only and creates no obligation in this session.

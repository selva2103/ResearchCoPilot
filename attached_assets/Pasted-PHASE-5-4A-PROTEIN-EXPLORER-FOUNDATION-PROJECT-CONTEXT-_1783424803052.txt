CRITICAL API NOTE — CANONICAL PROTEIN

Do NOT invent a canonical-protein heuristic. The canonical protein is
whichever protein is attached to the transcript already flagged
isCanonical: true (the MANE Select transcript, human genes only). If
TranscriptRecord.isCanonical is null (non-human gene), no protein in that
gene's list is canonical — do not guess based on length or accession order.

CRITICAL API NOTE — REVIEWED VS PREDICTED

Protein accession prefix maps deterministically:
- NP_ → status "Reviewed"
- XP_ → status "Predicted"
- anything else → "Other"

CRITICAL API NOTE — MOLECULAR WEIGHT

Molecular weight is NOT guaranteed in the NCBI protein GenPept response.
It appears only as an optional CALCULATED_MOL_WT qualifier in the feature
table. Model this field as optional/nullable. Do not fabricate it and do
not treat its absence as an error.

Before writing the parser: inspect a real GenPept response for
NP_000537.3 (TP53's canonical protein) to confirm actual field locations.
Build the parser against the observed schema, not assumed field names.

FILE 1: types/protein-record.ts

Create ProteinRecord interface with these exact fields:
- proteinAccession: string (e.g. NP_000537)
- proteinAccessionVersion: string (e.g. NP_000537.3)
- proteinName: string | null
- length: number | null (amino acids — use "aa" as the unit everywhere)
- molecularWeight: number | null (optional — see Critical API Note above)
- sequenceAvailable: boolean (true if the FASTA sequence is retrievable
  for this accession; set false if the accession is retired/unavailable,
  so the "Download FASTA" button can be disabled upfront in 5.4B rather
  than discovering the failure only on click)
- status: "Reviewed" | "Predicted" | "Other" (derived from NP_/XP_ prefix)
- sourceDatabase: "ncbi-refseq"
- isCanonical: boolean | null (true only if inherited from this protein's
  parent transcript's isCanonical; null for non-human genes; never
  false-positive)
- transcriptId: string (parent transcript's transcriptId, for traceback)
- proteinAccessionVersionSource: "transcript" (documents that this
  protein was never independently resolved)
- geneId: string
- geneSymbol: string
- organism: string
- ncbiProteinUrl: string (constructed from proteinAccessionVersion)

Document every field with JSDoc. Mark optional fields as optional.

FILE 2: lib/protein/fetch.ts

Function: fetchProteinSummaries(proteinAccessionVersions: string[])
- Batch protein summary retrieval using the minimum number of NCBI calls
  required. The implementation must first determine whether db=protein
  esummary accepts accession.version identifiers directly. If
  accession.version cannot be used reliably, resolve them internally to
  Protein UIDs using the smallest possible number of NCBI requests (e.g.
  batched esearch or elink), then perform the summary retrieval.
- This internal UID resolution must remain completely encapsulated
  inside the protein module. The rest of the application must continue
  to use only proteinAccessionVersion identifiers and must never depend
  on or expose NCBI UIDs.
- This is called once when a gene's transcript list is expanded, covering
  all proteins belonging to that gene's transcripts in the minimum number
  of NCBI calls — never one call per protein.
- Returned summaries must preserve the same order as the input
  proteinAccessionVersions array. The protein module must never reorder
  proteins independently of their parent transcript order.

Function: fetchProteinDetail(proteinAccessionVersion: string)
- Fetches full GenPept detail for ONE protein via
  efetch db=protein rettype=gp retmode=text
- Called only on demand, when a user expands a specific protein panel —
  not during the batched summary call above.

Both functions go through the existing ncbi_entrez_limiter. Use the
existing shared httpx/fetch client with UPSTREAM_TIMEOUT_SECONDS.

Cache strategy: cache summary, detail, and FASTA responses in the
existing Redis cache layer using these exact keys:
  protein:summary:{proteinAccessionVersion}
  protein:detail:{proteinAccessionVersion}
  protein:fasta:{proteinAccessionVersion}
Protein records are effectively immutable once versioned — cache
aggressively, no TTL shorter than what's already used for transcript
FASTA caching.

--------------------------------------------------

FILE 3: lib/protein/parser.ts

Function: parseProteinSummary(summaryResponse, transcriptRecord):
ProteinRecord
- Builds a ProteinRecord from the batched summary response for one
  protein, using the parent TranscriptRecord for isCanonical inheritance,
  geneId, geneSymbol, organism, and transcriptId.
- Apply NP_/XP_ → status mapping deterministically.
- isCanonical = transcriptRecord.isCanonical (direct inheritance, no
  independent logic).

Function: enrichWithDetail(proteinRecord, genPeptResponse): ProteinRecord
- Merges in proteinName, molecularWeight (if present), and length from
  the full GenPept detail fetch. Called only when detail is fetched
  on-demand (Step: user expands protein panel).

ProteinRecord is immutable after creation. enrichWithDetail must return a
NEW ProteinRecord object with merged fields rather than mutating the
object passed in. This matches the existing TranscriptRecord pattern and
avoids stale-reference bugs in the UI.

--------------------------------------------------

FILE 4: lib/protein/index.ts

Function: getProteinsForTranscripts(transcripts: TranscriptRecord[]):
Promise<ModuleResult<ProteinRecord>>
- Filters transcripts to only those with a non-null proteinAccession.
- Batches the summary fetch (FILE 2) across all filtered transcripts in
  one call.
- Returns ModuleResult<ProteinRecord> with:
  - status: "success" | "partial" | "empty" | "error"
  - data: ProteinRecord[]
  - count: data.length (never manually assigned)
  - executionTimeMs: measured
  - cached: boolean (true if served from Redis)
  - timestamp: ISO 8601

Function: getProteinDetail(proteinAccessionVersion: string):
Promise<ModuleResult<ProteinRecord>>
- Called when a single protein panel is expanded. Fetches detail (FILE 2),
  enriches (FILE 3), returns single-item ModuleResult.

--------------------------------------------------

ERROR HANDLING

- No proteinAccession on transcript → explicit "Non-coding transcript —
  no protein" state (Integration 1) — never a blank section, never an
  attempted fetch.
- Protein summary fetch fails → status "error", transcript row still
  visible, protein sub-section shows "Protein data temporarily
  unavailable."
- Protein detail fetch fails (e.g. retired accession) → error state in
  the protein sub-section only, with retry — transcript row and other
  transcripts unaffected.
- FASTA download fails → error state on that button, retry available,
  other transcripts/proteins unaffected.
- Never crash the transcript row or gene card because a protein
  operation failed.

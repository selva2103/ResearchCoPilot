SCOPE REFOCUS (per design Section 1)

5.4A produced correct raw protein data (accession, length, status, MW,
canonical flag) but no research context. This phase adds a derived,
researcher-facing interpretation layer on top of ProteinRecord — it
does NOT add new external data sources and does NOT modify how
ProteinRecord itself is fetched, parsed, or cached.

--------------------------------------------------

NON-NEGOTIABLE CONSTRAINTS (per design Sections 2, 4, 6, 9)

Do NOT:
- Add any new external API or database (no UniProt, GO, AlphaFold, PDB,
  InterPro, Pfam, Reactome, KEGG, drug targets). Everything in this
  phase derives from data already fetched in 5.4A (GenPept detail) and
  5.2 (Gene Explorer's already-fetched OMIM/disease association data).
- Make a new NCBI/OMIM API call to obtain Biological Importance data. If
  Gene Explorer's OMIM/disease association data is not already present
  in the current request's GeneRecord, omit Biological Importance
  entirely — do not fetch it fresh.
- Use an LLM call to generate the biological summary. Summary generation
  is structured extraction + formatting + sentence-joining from GenPept
  `DEFINITION`/`COMMENT` text only. No inference, no paraphrasing that
  changes meaning, no invented content to pad a sparse summary.
- Fabricate biological role chips or Biological Importance text when the
  underlying structured data doesn't support them. Omit the field
  instead.
- Retrofit Gene Explorer or Transcript Explorer onto the new
  `ResearchContext<T>` type. That generic type is defined for reuse
  later (Variant Explorer, 5.5+) but is NOT applied to any existing
  stable module in this phase.
- Modify 5.4A's ProteinRecord shape, its fetch/parse/cache logic, or its
  FASTA download mechanism.
- Build live PubMed-sourced Research Notes. The Research Notes field is
  a structural placeholder only in this phase — always empty/absent in
  the UI unless genuinely populated (which will not happen until a
  future phase).
- Build navigation-anchor/auto-scroll UI unless it adds no meaningful
  complexity — this is optional polish, not a requirement to satisfy.

FILE 1: types/research-context.ts

Create the generic interface and the protein-specific instantiation,
exactly as specified in the design (Section 8):

interface ResearchContext<T> {
  subject: T;
  summary: { text: string; source: string } | null;
  roleChips: { label: string; source: string }[];
  resolutionConfidence: "high" | "medium" | "low" | "ambiguous";
  annotationConfidence: "well-annotated" | "limited" | "unavailable";
  biologicalImportance: { text: string; source: string } | null;
  relationships: Record<string, string>;
  researchNotesPlaceholder: unknown[] | null;
}

Additionally define (per design Section 2's data model list):
- `ProteinResearchContext` = `ResearchContext<ProteinRecord>`, with the
  following protein-specific additions layered on top of the generic
  shape:
  - `canonicalExplanation: string` (the plain-language sentence — see
    FILE 2)
- Document every field with JSDoc, noting explicitly in the JSDoc for
  `ResearchContext<T>` that it is intentionally generic but, per this
  phase's scope, is currently only instantiated for Protein — not
  retrofitted onto Gene or Transcript.

VERSION AWARENESS RULE: Every identifier used to key, cache, or identify
a ProteinResearchContext must always use the full accession version
(e.g. NP_000537.3), never the bare accession (NP_000537). Do not strip
or normalize away the version anywhere in this phase's code.

FILE 2: lib/protein/research-context.ts

Implement pure derivation functions. None of these make new network
calls — all inputs are already-fetched data passed in as arguments.

Function: deriveSummary(genPeptCommentText, genPeptDefinitionText):
{ text: string; source: string } | null
- Structured extraction + formatting + sentence-joining ONLY. If input
  text is empty/sparse, return a shorter result or null — never pad with
  invented content. `source` must name the specific GenPept field(s)
  used (e.g. "RefSeq GenPept COMMENT"). Document the exact extraction
  rule used (which GenPept fields, what joining logic) in the final
  report.

Function: deriveRoleChips(genPeptFeatures/keywords): { label: string;
source: string }[]
- Extract short role labels from GenPept keyword/feature annotation
  where reliably present. Each chip's `source` must name the specific
  GenPept field it came from (e.g. "RefSeq GenPept KEYWORDS"). Return an
  empty array if nothing reliable is found — never guess a role.
  Extraction must be rule-based and data-driven from the GenPept
  response itself. Do NOT maintain a manually curated mapping (e.g. a
  hardcoded table like "TP53" → "Tumor suppressor") — that has the same
  scaling flaw already avoided elsewhere in this project (Phase R's gene
  list, disease keyword list).

Function: deriveCanonicalExplanation(proteinRecord, transcriptRecord):
string
- If proteinRecord.isCanonical is true: return a sentence stating this
  is the canonical protein via the MANE Select transcript.
- If proteinRecord.isCanonical is false: return a sentence stating this
  is a non-canonical isoform and naming the canonical accession if
  known from the same gene's protein set.
- If proteinRecord.isCanonical is null (non-human gene, per Phase R's
  null-not-false rule): return a sentence stating canonical designation
  does not apply to this organism — do not imply canonical/non-canonical
  status where MANE does not exist.

Function: computeAnnotationConfidence(genPeptDataCompleteness signals):
"well-annotated" | "limited" | "unavailable"
- Derived from how much structured GenPept data was actually available
  (e.g. presence/absence of COMMENT text, keyword/feature data, protein
  name). Document the exact rule used (what counts as
  well-annotated/limited/unavailable) in the final report — this must
  be a reasoned rule, not an arbitrary cutoff picked without
  justification.
- Internally, compute and log (but do NOT display in the UI) an evidence
  coverage signal — e.g. which of {COMMENT, DEFINITION, KEYWORDS,
  FEATURES} were present vs. absent for this protein, and the resulting
  fraction. This is not part of the public ProteinResearchContext type
  in this phase; it exists only as an internal/logged value that
  computeAnnotationConfidence's label is derived from, so future quality
  metrics have this data available without recomputation.

Function: mapResolutionConfidence(normalizedQueryConfidence: number,
ambiguous: boolean): "high" | "medium" | "low" | "ambiguous"
- Maps Phase R's numeric resolver confidence (and its ambiguous flag)
  to a researcher-facing label. Document the exact mapping rule used in
  the final report — this is a label translation of Phase R's existing
  evidence-based score, not a new confidence system.
- READ-ONLY RULE: this function must never modify
  NormalizedQuery.confidence or any other field on the NormalizedQuery
  object. It only reads and translates the value into a label.

Function: deriveBiologicalImportance(geneRecordOmimData): { text: string;
source: string } | null
- ONLY uses OMIM/disease association data already present on the
  GeneRecord passed in — never triggers a new fetch. If that data is
  absent or the current GeneRecord wasn't fetched with it, return null.
  The `source` field must name where the text came from (e.g. "Gene
  Explorer OMIM association").
- GRACEFUL DEGRADATION: if OMIM data exists on the GeneRecord but is too
  sparse/insufficient to form a genuine, specific statement, also return
  null — do NOT fall back to a generic filler sentence (e.g. "This gene
  is associated with disease"). A vague sentence is worse than no
  sentence; omission is always preferred over generic filler.

Function: buildRelationships(geneRecord, transcriptRecord, proteinRecord):
Record<string, string>
- Builds the Gene → Transcript → Protein → Species chain from data
  already in memory (geneId, geneSymbol, transcriptId, accessionVersion,
  proteinAccessionVersion, organism). No new API calls.

ProteinResearchContext must be immutable after creation, consistent with
the existing ProteinRecord/TranscriptRecord immutability pattern already
used in this codebase.

--------------------------------------------------

FILE 3: lib/protein/index.ts (extend, do not rewrite)

Function: getProteinResearchContext(proteinRecord, transcriptRecord,
geneRecord, normalizedQuery): Promise<ModuleResult<ProteinResearchContext>>
- Orchestrates the FILE 2 functions using data already available from
  the existing pipeline (ProteinRecord from 5.4A, TranscriptRecord from
  5.3, GeneRecord from 5.2, NormalizedQuery from Phase R). Makes zero
  new network calls.
- Cache the resulting ProteinResearchContext in the existing Redis cache
  layer using this exact key:
    researchcontext:protein:{proteinAccessionVersion}
  This is a separate key from 5.4A's protein:summary/detail/fasta keys,
  per the design's caching-separation rule (raw records vs. derived
  interpretation have different staleness properties).
- Returns ModuleResult<ProteinResearchContext> with status, data, count,
  executionTimeMs, cached, timestamp — same shape convention as existing
  ModuleResult usage elsewhere.

Do not modify the existing getProteinsForTranscripts or getProteinDetail
functions from 5.4A. This is a new, additive function only.

`researchNotesPlaceholder` must never be serialized as an empty array in
the cache. Store `null` when there is nothing to attach. An empty array
implies "checked, found nothing structured to show" versus null meaning
"no attachment yet" — future phases populating this field need that
distinction, and null is the cleaner, smaller cache payload.

IMMUTABILITY RULE

Once a ProteinResearchContext has been derived for a specific
ProteinRecord, it must never mutate the underlying ProteinRecord,
TranscriptRecord, GeneRecord, or NormalizedQuery. The Research Context
layer is strictly read-only and derived. Any future enrichment (PubMed,
UniProt, GO, Structures) must create a new derived object rather than
modifying existing cached records. This applies to every function in
FILE 2 and FILE 3 — none of them may write back to their input objects.

DOMAIN/UI SEPARATION RULE

ProteinResearchContext must remain a pure domain model with no
UI-specific formatting — no HTML, JSX, Markdown, color codes, icon
names, or badge-rendering strings anywhere in FILE 1, FILE 2, or FILE 3.
All presentation decisions (chip styling, confidence indicator visuals,
etc.) belong entirely to Integration 2's UI layer, which consumes the
plain data and decides how to render it.

LAZY LOADING RULE: Research Context must not trigger any derivation or
rendering work until the user explicitly expands this subsection —
consistent with the lazy-loading behavior established in 5.4A. Even
though the underlying derivation is local/computational rather than a
network call, the UX contract (nothing extra happens until the user

- Collapsed by default, consistent with 5.4A's lazy-load pattern.

  triggered by expanding this section):
  - Biological summary (`.text`, if present; omitted cleanly if null).
    Its `.source` is available for an on-demand provenance display (e.g.
    hover/tap), same treatment as role chip evidence — not shown inline
    by default.
  - Biological role chips — small badge/chip components, styled
    distinctly from but consistent with existing badge patterns (MANE
    badge, status badge). Each chip shows its evidence/source on
    hover/tap, not inline by default, so the card isn't visually
    cluttered.
  - Canonical/isoform explanation sentence
  - Two distinct confidence indicators (Resolution Confidence and
    Annotation Confidence) — visually distinguishable from each other;
    exact visual treatment (label, dot, badge) is an implementation
    choice, not fixed by this prompt.
  - Biological Importance (if present; omitted cleanly, no "coming
    soon" placeholder, if null)
  - Protein Relationships — a compact visual chain: Gene → Transcript →
    Protein → Species, using existing typography/badge patterns, no new
    component family.
  - Research Notes: not rendered at all in this phase (structural
    placeholder only in the data model; never shown as an empty section
    in the UI).
- Error state: if Research Context derivation fails but ProteinRecord
  succeeded, show the existing identifiers/FASTA panel normally and show
  only a quiet "Context unavailable" note in place of the Research
  Context section — never hide already-successful data.
- Empty states: optional fields (Biological Importance, role chips) are
  simply absent when no grounded/reliable data exists.
- Mobile: reuse existing responsive text-truncation patterns from Gene
  Explorer's summary section for the biological summary text.
- Navigation anchors: optional per the design — implement only if
  trivial; do not treat as a requirement.

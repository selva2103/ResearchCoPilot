CRITICAL API NOTE — CANONICAL TRANSCRIPT AND MANE SELECT

"Canonical transcript" is NOT a field that NCBI returns directly from 
EFetch or ESummary. You must NOT invent your own canonical-picking 
heuristic. Handle it as follows:

For HUMAN genes: MANE Select (Matched Annotation from NCBI and EMBL-EBI) 
is the official canonical transcript designation. It is available as a tag 
in the NCBI Gene EFetch response (look for MANE_Select in the transcript 
list). If a MANE Select transcript exists, mark it as canonical. If a 
MANE Plus Clinical transcript exists, mark it as such separately.

For NON-HUMAN genes: MANE does not exist. Do NOT apply a canonical label 
to non-human transcripts based on your own logic. Instead, set 
isCanonical: null and display nothing in the UI where "canonical" would 
appear — do not show "canonical" for a mouse or plant transcript just 
because it is the first result or longest sequence.

This distinction must be implemented correctly and verified in Step 9 
validation using both a human gene (TP53) and a non-human gene.

Before writing the parser: inspect real NCBI Gene EFetch responses for 
TP53 and BRCA1 to confirm the actual structure of the transcript list, 
the field name(s) used for MANE Select status, and the field name(s) used 
for transcript type (mRNA, ncRNA, etc.). Build the parser against the 
observed schema, not assumed field names.

GOAL

When a user searches TP53 and sees the Gene Explorer card, they should now 
be able to see a structured list of transcripts for that gene: accession, 
type, length, status, and whether it is the MANE Select canonical 
transcript (for human genes). No sequence retrieval, no downloads — just 
structured metadata that answers "what transcripts exist for this gene?"

STEP 1 — TranscriptRecord Interface

Create a strongly typed TranscriptRecord interface in 
types/transcript-record.ts. Fields:

Core fields:
- transcriptId (string) — NCBI transcript accession (NM_, NR_, XM_, XR_)
- accessionVersion (string) — full versioned accession e.g. NM_000546.6
- transcriptType - isProteinCoding (boolean) — derived deterministically from the
  transcript accession type and/or NCBI Gene annotation.
  NM_ and XM_ transcripts should normally be true.
  NR_ and XR_ transcripts should always be false.
  Do not infer beyond the annotation returned by NCBI.("mRNA" | "ncRNA" | "predicted_mRNA" | "predicted_ncRNA" 
  | "other") — derived from accession prefix per the rules below
- geneId (string) — parent NCBI Gene ID, linking back to GeneRecord
- geneSymbol (string) — parent gene symbol
- organism (string)
- transcriptLength (number | null) — length in nucleotides (nt), not bp 
  — use "nt" as the unit everywhere, not "bp"
- exonCount (number | null)
- status ("Reviewed" | "Validated" | "Provisional" | "Predicted" | 
  "Inferred" | "Model" | null) — use NCBI's actual RefSeq status values, 
  not a simplified binary
- isCanonical (boolean | null) — true only when MANE Select confirmed; 
  null for non-human genes (see Critical API Note above); never false-positive
- maneSelectAccession (string | null) — the MANE Select accession if 
  applicable
- manePlusClinical (boolean) — true if this transcript is also MANE Plus 
  Clinical
- sourceDatabase ("ncbi-refseq")
- ncbiTranscriptUrl (string) — constructed from accessionVersion

Accession prefix → transcriptType mapping (apply deterministically):
- NM_ → "mRNA" (curated)
- NR_ → "ncRNA" (curated)  
- XM_ → "predicted_mRNA"
- XR_ → "predicted_ncRNA"
- anything else → "other"

Document every field with JSDoc. Mark optional fields as optional.

STEP 2 — File Structure

lib/transcript/fetch.ts — fetch transcript list for a given NCBI Gene ID
lib/transcript/parser.ts — normalize NCBI response into TranscriptRecord[]
lib/transcript/index.ts — orchestrate and return ModuleResult<TranscriptRecord>

Keep the transcript module entirely separate from the gene module. It 
receives a geneId as input, not a raw query string — the Gene Explorer 
(Phase 5.2) resolves the gene first, then passes the geneId to the 
transcript module.

--------------------------------------------------

STEP 3 — Data Retrieval

The transcript list for a gene is available via NCBI Gene EFetch (db=gene) 
— the same EFetch call used by Gene Explorer (Phase 5.2) Path A. Do NOT 
make a separate ESearch for transcripts. Reuse the gene record response 
to extract the transcript list, or pass the already-fetched gene data 
into the transcript parser, rather than making a redundant NCBI call.

If the gene record is already available in the current request context 
(from Phase 5.2's fetch), use it directly. This avoids one unnecessary 
Entrez call per query and reduces rate-limit pressure.

Transcript extraction must reuse the exact Gene EFetch response already 
retrieved by Phase 5.2. Do NOT perform a second Gene EFetch within the 
same request. If the same gene is expanded multiple times during one 
session, reuse the already-parsed TranscriptRecord[] rather than 
re-parsing the Gene EFetch response. Document the caching strategy used 
(e.g. in-memory Map keyed by geneId) in the final report.

For each transcript in the list, retrieve only what is available in the 
gene record response for this phase — do NOT make individual EFetch calls 
per transcript to retrieve sequence data. That belongs to Phase 5.3B.

STEP 7 — Frontend: Transcript Explorer Card

Add a "Transcript Explorer" section within or below the Gene Explorer 
card. Reuse existing card style — do NOT redesign the application.

Display per transcript:
- Accession + version (e.g. NM_000546.6) — linked to NCBI
- Accession prefix badge (NM_, NR_, XM_, XR_) displayed as a small 
  distinct badge alongside the accession. This allows researchers to 
  immediately distinguish curated RefSeq transcripts from predicted 
  transcripts without expanding the row. Style it visually distinct from 
  the MANE Select and status badges — it represents the data source 
  class, not quality or canonicality.
- Transcript type label ("mRNA", "ncRNA", "Predicted mRNA", etc.)
- Length in nt (e.g. "2591 nt") — always show unit, never bare number
- Exon count (if available, else "not available")
- RefSeq status badge (Reviewed / Validated / Predicted / etc.)
- MANE Select badge — shown ONLY when isCanonical is true (human genes). 
  For non-human genes, this badge must NOT appear at all, not even 
  greyed out.
- MANE Plus Clinical badge — shown only when manePlusClinical is true

List ordering: MANE Select transcript first (if exists), then by 
transcriptType (curated before predicted), then by length descending.

Stub for Phase 5.3B (visible but not yet functional):
- "View Sequence" button — disabled, with tooltip "Available in next 
  update" — do NOT wire this to any functionality yet
- "Download FASTA" button — same treatment

Do NOT add expand/collapse panels in this phase. The transcript list is 
flat for now.

STEP 8 — Error Handling

Use ModuleResult consistently:
- No transcripts found → status: "empty", display "No RefSeq transcripts 
  found for this gene." — do not leave the section blank
- Transcript fetch failed → status: "error", gene card still visible, 
  transcript section shows "Transcript data temporarily unavailable."
- Partial transcript data → status: "partial", show what was retrieved

Never crash the gene card because transcripts failed.

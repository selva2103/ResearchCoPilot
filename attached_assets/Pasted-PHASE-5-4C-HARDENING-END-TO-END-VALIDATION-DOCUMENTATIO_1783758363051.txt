STEP 4 — ERROR-STATE TESTING

Confirm graceful, non-crashing behavior for each of the following,
across Gene Explorer, Transcript Explorer, Protein Explorer, and
Research Context:
1. Query with no matching gene at all → clear "no results" state, no
   crash, no partial/broken UI.
2. Ambiguous query with multiple candidates (e.g. "BRCA") → candidate
   list still renders correctly, selecting one still triggers the full
   chain correctly for that specific GeneID.
3. Non-coding transcript (NR_/XR_) → confirm "Non-coding transcript —
   no protein" state still displays correctly, Research Context section
   does not attempt to render for these.
4. A deliberately invalid/retired protein accession (if simulable) →
   confirm Protein Explorer and Research Context both show a handled
   error state with retry, not an unhandled exception.
5. Simulate a Research Context derivation failure (e.g. malformed
   GenPept response) while the underlying ProteinRecord succeeds →
   confirm identifiers/FASTA still display normally, only the Research
   Context section shows "Context unavailable," per the existing
   error-state design from 5.4B.
6. Rate limit hit during rapid downloads → confirm the existing
   "NCBI rate limit reached, please wait a moment" message (or
   equivalent) still surfaces correctly rather than a silent failure.

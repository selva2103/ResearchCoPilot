THE BUG

When a user searches a species-qualified query like:
  "mouse CD4", "mouse Trp53", "rat EGFR", "zebrafish Sox2"

The Gene Explorer currently returns the wrong organism's gene as the
primary result. Example: "mouse CD4" returns Rhea americana (or similar)
instead of Mus musculus CD4.

Root cause: The gene ranking step does not use the organism context
detected by the Query Resolver when sorting NCBI Gene ESearch results.
NCBI ESearch for db=gene returns results in its own relevance order,
which does not prioritize the organism the user specified. When the
resolver correctly identifies "mouse" as Mus musculus, that information
is not being passed to — or not being used by — the gene ranking step.

--------------------------------------------------

WHAT TO FIX

FIX 1 — Organism extraction from query

The resolver must reliably extract organism context from species-qualified
queries. Support at minimum these common prefixes and their NCBI Taxonomy
IDs:

  "mouse" / "murine"     → Mus musculus        (taxId: 10090)
  "rat"                  → Rattus norvegicus    (taxId: 10116)
  "zebrafish"            → Danio rerio          (taxId: 7955)
  "human" / "homo"       → Homo sapiens         (taxId: 9606)
  "fly" / "drosophila"   → Drosophila melanogaster (taxId: 7227)
  "worm" / "C. elegans"  → Caenorhabditis elegans (taxId: 6239)
  "yeast"                → Saccharomyces cerevisiae (taxId: 4932)
  "arabidopsis"          → Arabidopsis thaliana (taxId: 3702)
  "chicken"              → Gallus gallus         (taxId: 9031)

When an organism prefix is detected:
- Strip the organism prefix from the gene symbol before searching
  (e.g. "mouse CD4" → search for "CD4" with taxId filter 10090)
- Store the detected taxId in QueryResolution for downstream use
- Set confidence to HIGH (≥ 0.90) when both organism AND gene symbol
  are cleanly extracted

If no organism prefix is detected, behavior is unchanged from current.

FIX 2 — Pass organism context to NCBI Gene ESearch

When a taxId is available in QueryResolution, add it as a filter to the
NCBI Gene ESearch query:

  Instead of: esearch.fcgi?db=gene&term=CD4
  Use:        esearch.fcgi?db=gene&term=CD4[Gene Name]+AND+10090[Taxonomy ID]

This is NCBI's documented approach for organism-filtered gene searches.
Verify the exact field tag ([Taxonomy ID] vs [taxid]) against the actual
NCBI ESearch documentation or a live test query before implementing —
do not guess at the field name.

FIX 3 — Result ranking when multiple organisms are returned

Even with a taxId filter, ESearch may return results from multiple
organisms (due to NCBI's own ranking). Add a post-search ranking step:

Priority order for returned gene records:
  1. Exact taxId match to the requested organism
  2. Same genus (first word of scientific name matches)
  3. All other organisms

Within each priority tier, rank by:
  - "Reference" or "Representative" genome status first
  - Reviewed/Validated RefSeq status
  - Result position from NCBI (as tiebreaker)

The top result shown to the user must always be from the requested
organism when one was specified and a match exists in the results.

FIX 4 — Unqualified gene symbol queries (no organism prefix)

For queries without an organism prefix (e.g. plain "TP53", "EGFR"):
- Current behavior: return all organisms, Homo sapiens often ranked first
  by NCBI — this is acceptable and should be preserved
- Do NOT change this behavior
- Do NOT default unqualified queries to human — some users searching
  "TP53" may want non-human results and the multi-organism list serves them

FIX 5 — QueryResolution interface update

Add to QueryResolution (types/query-resolution.ts or equivalent):
- detectedOrganismTaxId: number | null
- detectedOrganismName: string | null  (e.g. "Mus musculus")
- strippedGeneQuery: string | null     (gene symbol after organism removal)

These fields allow downstream modules (Gene Explorer, and later Protein
Explorer) to use organism context without re-parsing the original query.

FIX 6 — Preserve organism context through the pipeline

When detectedOrganismTaxId is present, propagate it through the entire gene search pipeline:

QueryResolution → Gene Explorer → Gene search → Gene ranking

The detected organism context must remain attached to the request and be reused by downstream modules. Do not re-parse the original user query inside Gene Explorer or any downstream module.

All downstream ranking decisions must use the propagated detectedOrganismTaxId and detectedOrganismName from QueryResolution rather than attempting to infer the organism again.

This propagated organism context will also become the canonical source for Transcript Explorer, Protein Explorer (Phase 5.4), and future organism-aware modules.

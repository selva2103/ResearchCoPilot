   (pick one entity and stop) into: multi-entity extraction (detect all

ROOT CAUSE — WHY TRANSCRIPTS/PROTEINS GO MISSING

A gene card rendering successfully does not guarantee downstream modules
received a usable identifier. If the resolver returns only a gene symbol
or an ambiguous match — not a resolved NCBI GeneID — the transcript
module has no reliable key and effectively starts a new, less specific
search. This is why Transcript Explorer can show "No transcript" and
Protein Explorer can show "No protein" even when Gene Explorer displays a
seemingly valid gene.

The fix: the resolver becomes the single source of truth. It outputs one
structured object. Every downstream module consumes that same object —
none of them re-resolve the query independently.

FILE 1: types/normalized-query.ts

Create NormalizedQuery interface:

interface CandidateResolution {
  gene: { symbol: string; geneId: string | null } | null;
  organism: { name: string; taxId: string | null } | null;
  confidence: number;
}

interface NormalizedQuery {
  rawQuery: string;                    // original user input, always preserved
  gene: {
    symbol: string;
    geneId: string | null;             // NCBI GeneID — the stable key
    organismMatched: string | null;    // which organism this symbol was resolved for
  } | null;
  organism: {
    name: string;                      // e.g. "Mus musculus"
    taxId: string | null;
    matchedSynonym: string | null;     // e.g. "mouse" -> "Mus musculus"
  } | null;
  disease: {
    name: string;
  } | null;
  protein: {
    accession: string;
  } | null;
  confidence: number;                  // derived from entity agreement, see rules below
  candidates: CandidateResolution[] | null; // flat alternatives, never nested NormalizedQuery
  ambiguous: boolean;
  evidence: {
    source: "ncbi-gene" | "medgen" | "taxonomy" | "synonym";
    matchedValue: string;
    reason: string;
  }[];                                  // e.g. {source:"ncbi-gene", matchedValue:"TP53",
                                         // reason:"Official gene symbol"} — records WHY
                                         // each part of the resolution was made, for debugging
}

ARCHITECTURE NOTE — who consumes NormalizedQuery:
Only app/api/analyze/route.ts (the orchestrator) consumes the full
NormalizedQuery object directly. Each downstream module must NOT accept
NormalizedQuery as its input type. Instead:
- Gene Explorer accepts a resolved gene identifier (geneId if present,
  else symbol+organism as fallback) — derived from NormalizedQuery.gene
  by the orchestrator, then passed down as plain arguments/GeneRecord
  input, exactly as it already does today.
- Transcript Explorer accepts a geneId (from the resulting GeneRecord),
  exactly as it already does today.
- Protein Explorer accepts a proteinAccessionVersion (from the resulting
  TranscriptRecord), exactly as it already does today, unchanged from
  Phase 5.4A.
This keeps every module loosely coupled and independently reusable — the
resolver's output shape is the orchestrator's concern, not each module's.

FILE 2: Resolver changes (lib/resolver/ — locate existing files)

Bug 1 & 3 fix — Multi-entity extraction:
The resolver must extract ALL recognized entities from a single query,
not just one. "Trp53 Mus musculus" must populate BOTH gene AND organism
fields on the same NormalizedQuery — organism must never silently
replace/discard gene.

Bug 2 fix — Species-aware symbol resolution:
Gene symbols are organism-specific (TP53 human, Trp53 mouse, Tp53 rat).
When an organism is present in the query (explicit or inferred), resolve
the gene symbol against THAT organism's gene database, not a
default/generic one.

Organism detection must handle BOTH directions — do not replace one with
the other:
- Prefix pattern (organism stated before gene): "mouse Cd4" — this
  detection already exists and must be preserved.
- Suffix pattern (organism stated after gene): "Trp53 Mus musculus" —
  this detection must be added alongside the existing prefix detection.
Both patterns feed the same multi-entity extraction stage (Decision 2).

Bug 4 fix — Gene vs Disease priority (NO hardcoded gene list, NO
hardcoded disease keyword list):
This rule applies ONLY to the collision case — when a bare query string
matches BOTH a valid gene symbol for a recognized organism AND a disease
entity (e.g. "tp53" also matching a "TP53 polymorphism" disease entry).
In that specific collision case, prefer gene.symbol only when the query
consists solely of a recognized gene symbol with no additional
disease-specific context — e.g. "TP53" alone should resolve to gene, but
"TP53 mutation" or "TP53 polymorphism" should still allow disease.name
to populate alongside (or instead of) gene, since the query itself
carries disease-specific context beyond the bare symbol.

Do NOT detect "disease context" using a hardcoded keyword list (e.g.
"syndrome", "cancer", "disease"). That has the same maintainability flaw
as a hardcoded gene list. Instead, reuse the resolver's existing disease
classification/recognition logic (already built in 5.1.5 to support
disease queries generally) to determine whether the query contains a
genuine disease entity alongside the gene symbol. Do NOT introduce a new
mandatory MedGen/NCBI API call for every gene query solely to check for
disease context (per Decision 5 above) — this must reuse the existing
disease resolution flow and preserve current rate-limit behavior; no new
per-query Entrez/MedGen calls beyond what the resolver already makes
today. If the existing disease classifier finds no gene-symbol collision,
disease queries are handled exactly as they already are today — this
rule does not change disease resolution for non-colliding queries (e.g.
"hepatitis", "malaria", "diabetes") at all.

IDENTIFIER PRECEDENCE (applies across resolver and all modules)

When multiple identifier types are available for the same gene, always
prefer the highest-quality identifier in this order:
1. NCBI GeneID (most stable — always preferred when available)
2. RefSeq accession (e.g. NM_/NP_ accession, when GeneID is unavailable)
3. Ensembl Gene ID
4. Official gene symbol (organism-qualified)
5. Synonym / alternate symbol
6. Free text (last resort — lowest confidence)
Every module must use the highest-precedence identifier it has been
given, never falling back to a lower-precedence one when a
higher-precedence one is already available in the chain.

IDENTIFIER IMMUTABILITY RULE:
Once the resolver has produced a GeneID for the active pipeline, no
downstream module (Gene Explorer, Transcript Explorer, Protein Explorer,
Sequence Foundation, or any future module) may replace that GeneID with a
different identifier or perform an alternate resolution — unless the
resolver explicitly marked the query as ambiguous and a different
candidate is intentionally selected by the user. Downstream modules may
enrich the record with additional identifiers (RefSeq accession, Ensembl
ID, UniProt ID, etc.) but must never change the canonical GeneID
established by the resolver.

Bug 5 fix — Ambiguity (scoped down from original suggestion):
When multiple plausible interpretations exist with comparable confidence
(e.g. "BARC" matching multiple genes/organisms), populate the
`candidates` array and set `ambiguous: true`. Do NOT build a "did you
mean" disambiguation UI in this phase — this was previously and
deliberately deferred (see Phase 5.2 "Query Normalization" notes). The
resolver should surface the data; picking a default (highest-confidence)
candidate to proceed with, while making `candidates` available for a
future UI, is sufficient for this phase.

Bug 6 fix — Confidence scoring:
Confidence must be derived from resolver evidence and agreement between
entities — e.g. higher when a gene symbol is confirmed valid for the
stated/inferred organism, lower when entities are unconfirmed or
conflict. Do not hardcode fixed thresholds (e.g. "always 0.8" or "always
0.98"); the score should scale with how much evidence agrees, determined
algorithmically rather than fixed per query type.

Bug 7 fix — Context preservation:
"TP53 breast cancer" must populate BOTH gene.symbol = "TP53" AND
disease.name = "Breast Cancer" — not one or the other. This relies on
the same existing disease classifier referenced in the Bug 4 fix above,
running alongside gene detection rather than instead of it.

Bug 8 fix — Never silently discard the original query:
`rawQuery` is always preserved on every NormalizedQuery, unmodified,
regardless of what entities were or weren't extracted.

Bug 13 fix — Organism synonym normalization:
Normalize common synonyms to TaxID: "mouse"/"house mouse"/"Mus musculus"
→ taxId 10090; "human"/"man"/"Homo sapiens" → taxId 9606; and the other
organisms already supported (rat, zebrafish, fly, worm, yeast,
arabidopsis, chicken). Maintain this as a lookup table, not inline
string matching scattered through the code.

MULTI-GENE LAZY LOADING (explicit constraint for Bug 12)

Transcript and protein fetches must only occur after the user selects a
specific GeneID from a multi-gene result list. Do not prefetch
transcripts or proteins for any gene in the list before selection —
this significantly reduces unnecessary NCBI calls for ambiguous
multi-result queries (e.g. "BRCA" → 10 genes).
